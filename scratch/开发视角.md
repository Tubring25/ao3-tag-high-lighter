开发视角的功能拆分清单。

技术方案是：
	•	Chrome Extension Manifest V3
	•	前端用 TypeScript
	•	UI 可以先用 原生 DOM + CSS，不强依赖 React
	•	先做 桌面端 Chrome
	•	架构上预留移动端，但当前不实现移动端交互

⸻

一、先定开发范围

V1 / MVP 目标

做出一个可安装、可本地运行、可在 AO3 页面上生效的 Chrome 插件，支持：
	•	识别 AO3 页面中的 tag
	•	用户可通过页面内交互快速创建规则
	•	规则支持：
	•	喜欢：高亮 tag
	•	雷点：警示作品
	•	噪音：弱化 tag
	•	强雷点：折叠作品
	•	支持三种匹配方式：
	•	精确
	•	包含
	•	通配符
	•	支持三类 tag：
	•	relationship
	•	character
	•	freeform
	•	插件弹窗可查看当前页命中情况
	•	规则存本地，可管理、编辑、删除、启停

⸻

二、建议的整体技术架构

1. Extension 层

负责 Chrome 插件本身：
	•	manifest.json
	•	background/service worker
	•	popup
	•	options page
	•	storage

2. Content Script 层

注入 AO3 页面，负责：
	•	识别页面中的 tag / works
	•	执行规则匹配
	•	渲染样式变化
	•	挂载 hover 按钮和交互菜单

3. Core 核心逻辑层

和页面 / 插件 UI 解耦，负责：
	•	规则模型
	•	匹配引擎
	•	冲突优先级
	•	命中结果结构化输出

4. UI 层

分两块：
	•	页面内小按钮 + 菜单
	•	popup / options 管理界面

四、数据模型先定死

这一步很重要。
你后面很多功能能不能稳，取决于这里是否抽得干净。

⸻

4.1 Rule 规则模型

type RuleAction = "highlight" | "warn" | "mute" | "hideWork";

type MatchMode = "exact" | "contains" | "wildcard";

type TagCategory = "relationship" | "character" | "freeform" | "all";

interface Rule {
  id: string;
  label?: string;
  pattern: string;
  action: RuleAction;
  matchMode: MatchMode;
  category: TagCategory;
  enabled: boolean;
  createdAt: number;
  updatedAt: number;

  // 预留字段
  note?: string;
  groupId?: string | null;
  source?: "manual" | "quickAdd";
}


⸻

4.2 Settings 设置模型

interface Settings {
  extensionEnabled: boolean;
  hoverButtonEnabled: boolean;
  showToast: boolean;
  hideWorkMode: "collapse" | "hide";
  enableOnWorkDetailPage: boolean;
}

MVP 建议默认：

const defaultSettings: Settings = {
  extensionEnabled: true,
  hoverButtonEnabled: true,
  showToast: true,
  hideWorkMode: "collapse",
  enableOnWorkDetailPage: true,
};


⸻

4.3 页面解析结果模型

interface ParsedTag {
  id: string;
  text: string;
  normalizedText: string;
  category: "relationship" | "character" | "freeform";
  element: HTMLElement;
  workId: string;
}

interface ParsedWork {
  id: string;
  element: HTMLElement;
  tags: ParsedTag[];
}


⸻

4.4 命中结果模型

interface TagMatch {
  tagId: string;
  ruleId: string;
  action: RuleAction;
}

interface WorkMatchSummary {
  workId: string;
  matchedRuleIds: string[];
  hasWarn: boolean;
  hasHideWork: boolean;
}

interface MatchResult {
  tagMatches: TagMatch[];
  workSummaries: WorkMatchSummary[];
}

五、模块拆分清单

下面按模块拆。

⸻

模块 A：Manifest 与基础插件壳

A1. 建立插件基础工程

任务
	•	建立 MV3 项目
	•	配置 TypeScript
	•	配置打包工具（Vite / esbuild / webpack 任一）
	•	输出 manifest、content script、popup、options、background

验收标准
	•	插件能在 Chrome 开发者模式下正常加载
	•	popup 能打开
	•	content script 能注入 AO3 页面
	•	background worker 正常运行

⸻

A2. 配置最小权限

任务

在 manifest 中配置：
	•	storage
	•	scripting
	•	host_permissions：
	•	https://archiveofourown.org/*
	•	如有需要再加 https://archive.transformativeworks.org/*

验收标准
	•	不使用 <all_urls>
	•	插件安装提示权限可接受
	•	在 AO3 页面可正常运行

⸻

A3. 消息通信基础

任务

定义 popup / options / content / background 之间的消息类型：
	•	获取规则
	•	获取设置
	•	更新规则
	•	通知页面重算
	•	获取当前页统计

验收标准
	•	能从 popup 读到 content script 返回的当前页信息
	•	更新规则后 content script 可收到并刷新页面表现

⸻

模块 B：AO3 页面解析器

这是核心模块之一。

⸻

B1. 页面类型识别

任务

判断当前是否是 AO3 支持页面：
	•	works 列表页
	•	search results
	•	bookmarks list
	•	fandom listing
	•	single work detail

实现建议

先做简单判断：
	•	URL 模式
	•	DOM 是否存在 AO3 的作品列表容器 / tag 容器

验收标准
	•	非 AO3 页面不执行
	•	AO3 支持页面能进入解析流程

⸻

B2. 作品卡片识别

任务

从页面 DOM 中找到“单篇作品”的容器元素。

输出

ParsedWork[]

风险

不同 AO3 页面结构可能略有差异，不能把选择器写得太死。

验收标准
	•	在列表页能正确识别多篇作品
	•	在详情页能识别当前单篇作品为一个 work

⸻

B3. tag 提取

任务

对每个作品提取 tag，并识别所属分类：
	•	relationship
	•	character
	•	freeform

输出

每个 ParsedWork 包含 tags: ParsedTag[]

关键点

必须保留：
	•	原始文本
	•	规范化文本
	•	DOM 引用
	•	所属 workId

验收标准
	•	页面中 tag 数量与 DOM 实际一致
	•	category 分类基本正确
	•	每个 tag 能反向找到所属作品

⸻

B4. 文本规范化

任务

对 tag 文本做统一处理：
	•	trim
	•	多空格归一
	•	大小写策略统一
	•	可选特殊字符规范化

目的

避免“同一个 tag 因空格/大小写问题匹配失败”。

验收标准
	•	同样内容的 tag，不因大小写 / 空格轻易失配

⸻

B5. DOM 变化监听

任务

使用 MutationObserver 监听 AO3 页面变化：
	•	局部刷新
	•	懒加载
	•	页面局部重绘

目的

保证新增内容也会被识别和渲染。

验收标准
	•	页面发生 DOM 变化后，能自动重新解析并应用规则
	•	不出现明显重复渲染 / 性能爆炸

⸻

模块 C：规则引擎

这是另一个核心模块。

⸻

C1. 精确匹配

任务

实现：

exactMatch(rule.pattern, tag.normalizedText)

验收标准
	•	完全一致时命中
	•	仅部分相同不命中

⸻

C2. 包含匹配

任务

实现：

containsMatch(rule.pattern, tag.normalizedText)

验收标准
	•	子串存在时命中
	•	边界行为稳定

⸻

C3. 通配符匹配

任务

支持 *：
	•	Alternate Universe - *
	•	*Slow Burn*

实现建议

转换成正则，但注意转义普通字符。

关键函数

wildcardToRegExp(pattern: string): RegExp

验收标准
	•	* 可匹配任意长度字符串
	•	其他字符按字面匹配
	•	规则不被错误解释成完整正则语法

⸻

C4. 类别过滤

任务

规则命中前，先判断 tag category 是否匹配：
	•	rule.category = all → 全部可匹配
	•	否则必须和 tag.category 一致

验收标准
	•	relationship 规则不会误打到 freeform

⸻

C5. 规则启用状态

任务

只对 enabled === true 的规则参与计算。

验收标准
	•	禁用规则后页面效果立即消失

⸻

C6. 规则优先级

任务

实现固定优先级：
	1.	hideWork
	2.	warn
	3.	highlight
	4.	mute

场景

同一 tag 或作品同时命中多个规则时，决定最终表现。

验收标准
	•	hideWork 命中后作品优先折叠
	•	warn 不会被 highlight 覆盖掉作品级警示
	•	tag 级别样式不互相打架

⸻

C7. 规则计算入口

任务

实现统一入口：

evaluateRules(parsedWorks: ParsedWork[], rules: Rule[]): MatchResult

输出
	•	tagMatches
	•	每个 work 的汇总状态

验收标准
	•	给定解析结果和规则后，输出稳定、可预测的命中结果

⸻

模块 D：页面渲染器

负责把命中结果真正画到页面上。

⸻

D1. tag 高亮

任务

对 action = highlight 的 tag 添加样式类名，例如：
	•	ao3-ext-tag-highlight

验收标准
	•	喜欢的 tag 一眼可见
	•	样式不会把原页面搞崩

⸻

D2. tag 弱化

任务

对 action = mute 的 tag 添加弱化样式。

验收标准
	•	tag 仍可见
	•	但视觉权重明显低于普通 tag

⸻

D3. 作品警示

任务

对命中 warn 的作品卡片添加作品级样式：
	•	边框
	•	轻背景
	•	角标/小提示（可选）

验收标准
	•	用户能快速意识到这篇作品有雷点
	•	不隐藏 tag 文本本身

⸻

D4. 作品折叠 / 隐藏

任务

对命中 hideWork 的作品执行：
	•	默认 collapse
	•	可选 hide

推荐先做 collapse

折叠时保留占位条，示例内容：
	•	“此作品已折叠（命中 2 条规则）”
	•	“展开查看”

验收标准
	•	折叠稳定
	•	用户可恢复查看
	•	不会导致页面布局大面积错乱

⸻

D5. 清理旧样式

任务

每次重算时，先清理之前注入的 class / 占位元素 / data 标记。

目的

防止重复渲染、样式堆叠、同一作品被多次包装。

验收标准
	•	多次重新计算后 DOM 不脏
	•	不会出现重复按钮、重复折叠条

⸻

D6. 当前页统计计算

任务

根据命中结果得出：
	•	高亮 tag 数
	•	弱化 tag 数
	•	警示作品数
	•	折叠作品数
	•	命中的规则 ID 列表

验收标准
	•	popup 能正确显示当前页统计

⸻

模块 E：页面内交互

这是 MVP 的体验核心。

⸻

E1. hover 触发机制

任务

当鼠标进入 tag 区域时，显示一个小操作按钮。

关键点
	•	默认不显示
	•	不遮挡文字
	•	移开时消失
	•	同一时刻只保留一个活动入口更稳

验收标准
	•	用户能顺利发现入口
	•	页面不会因为大量 tag 同时 hover 而闪烁或卡顿

⸻

E2. 小按钮组件

任务

实现可复用的轻量按钮：
	•	定位到当前 tag 附近
	•	点击弹出菜单

验收标准
	•	在常见 tag 长度和布局下位置合理
	•	不会遮住相邻 tag

⸻

E3. 操作菜单

任务

点击按钮后弹出菜单项：
	•	标为喜欢
	•	标为雷点
	•	弱化显示
	•	隐藏命中作品

可选：
	•	查看已有规则
	•	取消

验收标准
	•	菜单打开/关闭行为稳定
	•	点击菜单项能正确创建规则

⸻

E4. 快速加规则逻辑

任务

从当前 tag 直接生成一条规则，默认：
	•	pattern = 当前 tag 原文
	•	action = 用户选择
	•	matchMode = exact
	•	category = 当前 tag 分类
	•	source = quickAdd

验收标准
	•	点一次即可生成规则
	•	页面立即重算并体现效果

⸻

E5. Toast / 操作反馈

任务

操作成功后显示轻量提示：
	•	已加入“喜欢”
	•	已加入“雷点”
	•	已折叠命中作品规则

验收标准
	•	用户知道操作成功
	•	提示不过度打扰

⸻

E6. 页面内状态更新

任务

用户添加规则后，不刷新页面，直接：
	•	更新本地规则
	•	重新运行规则引擎
	•	重绘页面表现

验收标准
	•	操作是即时的
	•	不需要用户手动刷新页面

六、开发优先级排序

下面给你一个比较现实的优先级。

⸻

P0：必须先做

这些不做，插件跑不起来。
	1.	Manifest + 基础工程
	2.	AO3 页面识别
	3.	作品 / tag 解析
	4.	规则数据模型
	5.	精确 / 包含 / 通配符匹配
	6.	规则优先级
	7.	页面样式渲染
	8.	本地存储
	9.	hover 小按钮
	10.	菜单快速加规则
	11.	popup 当前页统计
	12.	options 基础规则管理

⸻

P1：建议一起做

这些不做也能跑，但体验会明显差。
	1.	Toast 反馈
	2.	折叠作品占位条
	3.	规则启停
	4.	数据校验

七、推荐的开发顺序

这是最重要的部分之一。

⸻

第 1 阶段：跑通最小闭环

目标：页面识别 + 规则生效

做什么
	•	插件壳
	•	content script 注入
	•	AO3 tag 解析
	•	规则引擎
	•	高亮 / 警示 / 弱化 / 折叠渲染
	•	本地 storage 写死几条测试规则

完成标志

不用任何 UI，打开 AO3 页面就能看到规则效果。

⸻

第 2 阶段：做页面内交互

目标：用户能从 tag 上快速加规则

做什么
	•	hover 小按钮
	•	菜单
	•	quick add rule
	•	页面即时重算
	•	toast

完成标志

用户不进设置页，也能从页面直接建立规则。

⸻

目标：规则可维护

做什么
	•	popup 当前页统计
	•	options 规则列表
	•	手动新增 / 编辑 / 删除 / 启停

完成标志

用户已有规则可以完整维护。

⸻

第 4 阶段：补稳定性

目标：能实际内测

做什么
	•	清理旧状态
	•	DOM 变化监听
	•	节流
	•	错误兜底
	•	简单设置页

完成标志

可给真实用户装起来用。

九、建议先不纠结的实现问题

这些很容易把时间拖住，MVP 阶段先别深挖。
	1.	不要一开始做 React
	•	popup / options 用原生都够
	•	页面内菜单更适合轻量 DOM 实现
	2.	不要一开始做复杂选择器抽象
	•	先用清晰直白的 AO3 parser
	•	后面再重构
	3.	不要一开始做复杂规则表达式
	•	先把 exact / contains / wildcard 做稳
	4.	不要一开始做跨浏览器
	•	先只跑 Chrome
	5.	不要一开始做云同步
	•	本地 local storage 足够验证产品

十、我建议你接下来立刻落下来的三个文档

如果你准备开始写代码，下一步最该补的是这三份：

1. types.ts

把 Rule、Settings、ParsedTag、ParsedWork、MatchResult 全写出来。

2. ao3Parser.ts

先验证你能不能稳定从页面中拿到：
	•	作品容器
	•	各类 tag
	•	tag 对应 DOM

3. ruleEngine.ts

给 parser 输出喂几条写死规则，看看匹配和结果聚合是否正确。

只要这三件事跑通，后面的 hover、popup、options 都只是往上搭。
