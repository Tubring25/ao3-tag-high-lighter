
AO3 Tag 插件规划（桌面端优先，移动端架构预留）

0. 产品一句话定义

一个运行在 桌面端 Chrome 的 AO3 辅助插件。
它允许用户基于自己的 tag 规则，对 AO3 页面中的作品和 tag 进行：
	•	喜欢：高亮
	•	雷点：警示
	•	噪音：弱化
	•	强雷点：折叠 / 隐藏作品

目标不是改造 AO3，也不是做推荐系统，而是：

帮助用户更快扫 tag、更快做阅读决策、更稳定地管理个人偏好。

⸻

1. 产品定位

1.1 解决的问题

AO3 用户在浏览列表页时，经常会遇到这些问题：

1. tag 太多，扫读成本高

一篇作品常常带大量 relationship / character / freeform tag。
用户真正关心的只有一部分，但需要从很长的 tag 串中快速识别。

2. 容易漏看关键信息

有些 tag 对用户来说是决定性的：
	•	喜欢的 trope
	•	明确雷点
	•	作品风格或走向
	•	某些特定 AU / dynamic

如果列表里一眼看不到，就容易误点、误读。

3. 现有解决方式要么太原始，要么门槛高

用户通常只能：
	•	手动扫 tag
	•	用 AO3 自带搜索过滤
	•	依赖 userscript / site skin / 自己折腾样式

这些方式要么操作重，要么不够即时，要么不够产品化。

⸻

1.2 产品核心价值

这个插件的价值不是“能高亮 tag”本身，而是三点：

1. 加速决策

用户在列表页能更快判断“这篇值得点进去吗”。

2. 个性化视图

同样的 AO3 页面，不同用户看到的是适合自己偏好的“局部增强版”。

3. 低门槛维护规则

看到某个 tag，就能快速纳入自己的偏好体系，而不是去改脚本或手输一大堆配置。

⸻

1.3 用户类型

第一阶段优先服务这几类用户：

A. 强依赖 tag 决策的用户

点不点进去主要看 pairing、trope、warning、AU、freeform。

B. 有明确雷点的用户

需要快速发现风险，或者直接折叠命中强雷点的作品。

C. 浏览量大、扫列表很多的用户

长时间刷结果页、fandom 页、bookmark 页，希望减少认知疲劳。

D. 已经愿意做轻度个性化设置的用户

愿意维护十几条到几十条规则，不排斥插件配置。

⸻

2. 产品边界

2.1 当前阶段做什么

当前阶段只做：
	•	桌面端 Chrome
	•	当前页面增强
	•	本地规则存储
	•	本地匹配和显示处理
	•	规则可视化管理

⸻

2.2 当前阶段不做什么

以下内容 不在 MVP：
	•	移动端交互支持
	•	云同步
	•	登录系统
	•	AI 推荐
	•	自动翻页抓取
	•	跨页批量分析整站内容
	•	自动收藏 / 自动订阅 / 自动评论等自动化行为
	•	大规模 UI 改版

⸻

2.3 方案 B 落地原则

既然采用方案 B，那么设计上必须遵守：

1. 规则层与交互层分离

规则模型不依赖 hover。
以后移动端可以换成 tap / long press，而底层规则无需重写。

2. 匹配引擎与展示层分离

命中逻辑独立。
将来即使想支持不同浏览器、不同交互方式，也不需要重写匹配核心。

3. 页面注入逻辑与规则管理界面分离

内容脚本负责识别和标注；插件弹窗或设置页负责管理规则。

⸻

3. MVP 总目标

MVP 不是“尽量多做功能”，而是验证这几个关键点：
	1.	用户是否真的愿意建立规则
	2.	四类规则是否足够表达主要需求
	3.	悬停后轻量操作入口是否顺手
	4.	当前页即时反馈是否清晰
	5.	这个产品是否值得长期装着

⸻

4. MVP 功能总览

MVP 建议分成四个模块：
	1.	页面识别与命中
	2.	规则添加与管理
	3.	结果展示与交互反馈
	4.	本地存储与基础设置

⸻

5. MVP 详细功能

⸻

5.1 页面识别与命中模块

这是 MVP 的核心底层。

5.1.1 支持页面范围

首批支持这些 AO3 页面：
	•	搜索结果页
	•	Tag 浏览 / listing 页
	•	Fandom 作品列表页
	•	Bookmarks 列表页
	•	Works 列表型页面
	•	单篇作品详情页中的 tag 区域

为什么先做这些

因为用户的主要决策场景发生在：
	•	“看一堆作品卡片时”
	•	“点进去前”
	•	“快速筛选候选文时”

单篇作品详情页虽然也有价值，但主要是为了保持体验一致。

⸻

5.1.2 识别对象

插件需要识别页面上的：
	•	单个 tag 文本
	•	tag 类型
	•	所属作品卡片 / 作品容器
	•	同一作品下有哪些 tag 命中了哪些规则

⸻

5.1.3 支持的 tag 类型

MVP 至少区分：
	•	Relationship
	•	Character
	•	Freeform

未来可以考虑补充其他分类，但 MVP 先聚焦这三类。

⸻

5.1.4 匹配方式

MVP 支持三种：

1. 精确匹配
tag 完全等于规则文本才算命中。
适合对特定 tag 精准控制。

2. 包含匹配
只要 tag 文本包含该关键词即可命中。
适合泛搜索，但有误伤风险。

3. 通配符匹配
支持 * 作为任意字符占位。
适合批量覆盖一类 tag，而不是每条单独配置。

⸻

5.1.5 匹配作用范围

每条规则可以选择：
	•	仅 relationship 生效
	•	仅 character 生效
	•	仅 freeform 生效
	•	对全部 tag 生效

⸻

5.1.6 匹配优先级

MVP 必须有一个简单但明确的优先级逻辑，避免同一 tag 或同一作品被多种规则命中时混乱。

建议优先级：
	1.	隐藏作品
	2.	警示作品
	3.	高亮 tag
	4.	弱化 tag

解释：
	•	强雷点最优先，因为它影响整个作品是否展示
	•	警示次之，因为它影响作品整体风险呈现
	•	高亮和弱化只影响 tag 视觉层级

⸻

5.1.7 冲突处理规则

例如：
	•	同一 tag 同时命中“喜欢”和“弱化”
	•	同一作品同时命中“喜欢”tag 和“隐藏作品”tag

建议 MVP 简化成：

tag 级别冲突
优先使用优先级更高的规则表现，不叠加混乱样式。

作品级别冲突
只要命中“隐藏作品”，作品直接折叠；
若未命中隐藏但命中“雷点”，则作品进入警示状态。

⸻

5.2 规则系统模块

⸻

5.2.1 四类规则定义

MVP 采用你确认过的四类设计。

A. 喜欢（Highlight）
命中后，对对应 tag 做高亮。
目的：让用户一眼看到喜欢的元素。

B. 雷点（Warn）
命中后，不隐藏 tag 本身，而是对整篇作品卡片做警示。
目的：强化避雷，而不是削弱信息。

C. 噪音（Mute）
命中后，对对应 tag 做弱化显示。
目的：降低视觉干扰，让重要信息更突出。

D. 强雷点（Hide Work）
命中后，折叠或隐藏整篇作品。
目的：减少用户反复看到明确不想看的内容。

⸻

5.2.2 每条规则的数据结构

每条规则至少包含：
	•	规则 ID
	•	规则名称（可自动生成，也可用户编辑）
	•	目标文本
	•	匹配方式（精确 / 包含 / 通配符）
	•	作用类型（喜欢 / 雷点 / 噪音 / 隐藏作品）
	•	作用范围（relationship / character / freeform / all）
	•	是否启用
	•	创建时间
	•	更新时间

建议额外预留字段：
	•	备注
	•	分组 ID
	•	命中次数统计
	•	来源（手动输入 / 页面快速添加）

这些字段未来扩展会用到。

⸻

5.2.3 规则创建方式

MVP 支持两种：

1. 页面内快速添加
这是主入口。

用户在页面 tag 上悬停时，出现一个小按钮。
点击后弹出菜单，选择：
	•	标为喜欢
	•	标为雷点
	•	弱化显示
	•	隐藏命中作品

系统自动带入该 tag 文本，并默认匹配方式为“精确匹配”。

2. 插件弹窗 / 设置页手动添加
适合高级用户和批量维护。

可手动输入：
	•	tag 文本
	•	匹配方式
	•	类型
	•	作用范围

⸻

5.2.4 规则编辑能力

MVP 中每条规则至少要能：
	•	启用 / 禁用
	•	修改匹配文本
	•	修改匹配方式
	•	修改规则类型
	•	修改作用范围
	•	删除

⸻

5.2.5 规则管理界面

MVP 不需要特别复杂，但必须实用。

建议至少有：
	•	全部规则列表
	•	按规则类型筛选
	•	搜索规则文本
	•	查看规则是否启用
	•	快速编辑
	•	删除规则

⸻

5.2.6 导入 / 导出

这个我建议 纳入 MVP 尾部或 V1.1。
如果工期很紧，可以稍后放。

最小形式就是：
	•	导出 JSON
	•	导入 JSON

目的不是炫技，而是：
	•	方便备份
	•	方便迁移浏览器
	•	方便以后扩展规则模板

⸻

5.3 页面交互模块

⸻

5.3.1 tag 上的轻量入口

这是 MVP 的核心交互。

默认状态
tag 正常显示，不打扰用户阅读。

hover 状态
当鼠标悬停到某个 tag 上时，tag 旁边出现一个轻量小按钮。

按钮要求：
	•	体积小
	•	不遮挡 tag 文本
	•	风格尽量贴近 AO3 页面，不突兀
	•	平时不显示，减少干扰

⸻

5.3.2 点击菜单内容

点击小按钮后，弹出菜单：
	•	标为喜欢
	•	标为雷点
	•	弱化显示
	•	隐藏命中作品
	•	查看相关规则（可选）
	•	取消 / 关闭

MVP 可以先不做“查看相关规则”，但有余力建议做。

⸻

5.3.3 操作后反馈

必须有即时反馈，不然用户会不确定是否成功。

最小反馈包括：
	•	当前 tag 或当前作品立刻变化
	•	页面上出现轻量提示，比如“已加入雷点规则”
	•	可撤销更好，但不是 MVP 必须

⸻

5.3.4 命中后的视觉表现

喜欢
tag 高亮显示。
要求：清晰但不过分刺眼。

雷点
整篇作品卡片出现明显但不夸张的警示样式。
例如：
	•	边框强调
	•	轻微背景色提示
	•	警示标识

关键点：不要隐藏 tag 本身。

噪音
tag 降低视觉权重。
例如：
	•	降低对比度
	•	减弱颜色
	•	不删除、不完全不可见

隐藏作品
整篇作品卡片折叠或隐藏。
MVP 建议优先做“折叠”而不是彻底消失。

因为折叠更安全，用户仍可展开确认，不容易造成“为什么这篇没了”的困惑。

⸻

5.3.5 折叠作品的样式

如果命中“隐藏作品”，建议先显示一个折叠占位条，内容类似：
	•	已折叠 1 篇命中规则的作品
	•	展开查看

这样更稳，不容易引起误解。

⸻

5.3.6 当前页命中总览

插件弹窗里建议有一个“当前页命中”模块，显示：
	•	当前页命中了多少条规则
	•	有多少篇作品被警示
	•	有多少篇作品被折叠
	•	本页启用了哪些规则

这个功能价值很高，因为它能解释页面为什么变成现在这样。

⸻

5.4 插件弹窗模块

弹窗是轻管理入口，不是完整后台。

⸻

5.4.1 弹窗首页建议包含
	•	插件开关
	•	当前页命中统计
	•	快速查看最近添加的规则
	•	跳转到完整规则管理页

⸻

5.4.2 当前站点状态

弹窗中应明确显示：
	•	当前页面是否已启用插件
	•	当前页面识别到了哪些可处理内容
	•	当前是否有折叠作品 / 警示作品

⸻

5.4.3 全局开关

至少需要：
	•	全局启用 / 禁用插件
	•	仅当前页面暂时停用（可选）
	•	仅当前站点启用（默认）

⸻

5.5 设置与存储模块

⸻

5.5.1 本地存储

MVP 规则和设置全部保存在本地。

包括：
	•	规则列表
	•	全局开关
	•	样式偏好
	•	最近使用记录（可选）

⸻

5.5.2 基础设置项

MVP 建议提供这些设置：
	•	是否启用 hover 按钮
	•	是否显示操作反馈提示
	•	折叠作品时默认是“隐藏”还是“显示折叠占位”
	•	高亮 / 警示 / 弱化样式强度（简单版）
	•	是否在单篇作品页生效

⸻

5.5.3 样式适配

MVP 至少要考虑：
	•	AO3 默认主题下可读
	•	深色 / 反转风格下不至于失效
	•	样式尽量不与页面原生 CSS 严重冲突

不要求一开始做完整主题适配，但必须避免“颜色完全看不清”。

⸻

6. MVP 用户流程

⸻

6.1 首次使用流程
	1.	用户安装插件
	2.	打开 AO3 页面
	3.	插件开始识别 tag
	4.	用户 hover 到某个 tag
	5.	出现小按钮
	6.	点击按钮并选择规则类型
	7.	页面立刻生效
	8.	插件弹窗能看到新增规则和当前页命中情况

⸻

6.2 日常使用流程
	1.	用户像平常一样浏览 AO3
	2.	喜欢的 tag 自动高亮
	3.	雷点作品自动警示
	4.	强雷点作品自动折叠
	5.	不重要的 tag 自动弱化
	6.	用户遇到新 tag 时可随手加入规则

⸻

6.3 规则管理流程
	1.	用户打开插件弹窗或设置页
	2.	查看现有规则列表
	3.	搜索 / 编辑 / 禁用 / 删除
	4.	调整规则后返回 AO3 页面
	5.	页面实时刷新表现

⸻

7. MVP 非目标

这部分必须写清楚，否则范围会失控。

7.1 暂不支持移动端交互

不做 touch 菜单，不做长按方案，不做手机浏览器兼容。

7.2 暂不做智能推荐

不分析阅读历史，不预测用户喜好。

7.3 暂不做账号系统

不做登录，不做云端配置同步。

7.4 暂不做重度自动化

不自动操作 AO3 的收藏、评论、订阅等功能。

7.5 暂不做跨页批量扫描

不后台自动翻页，不抓取多页内容做大规模聚合。

⸻

8. MVP 成功标准

MVP 上线后，不是看“有没有做出来”，而是看有没有实际使用价值。

建议重点看这些指标：

8.1 激活指标
	•	安装后创建第一条规则的人占比
	•	安装后创建三条以上规则的人占比

8.2 使用指标
	•	每次 AO3 浏览会话中规则的平均命中数
	•	有多少用户会使用页面内快速添加
	•	有多少用户会进入规则管理页

8.3 留存指标
	•	第三天还有操作的比例
	•	一周后仍启用插件的比例
	•	用户是否持续新增规则

8.4 反馈指标
	•	用户是否觉得“扫 tag 更快”
	•	用户是否觉得“避雷更稳”
	•	用户是否觉得“不会打扰阅读”

⸻

9. 后续扩展计划

下面是比较合理的扩展方向，建议按阶段推进。

⸻

Phase 1：V1.1 优化版（MVP 后最优先）

这是最值得做的增强层。

9.1 规则分组

用户可以建立规则组，例如：
	•	喜欢的 trope
	•	强雷点
	•	某个 fandom 专用规则
	•	某个 pairing 专用规则

价值：
	•	规则变多后更容易管理
	•	为以后移动端简化管理打基础

⸻

9.2 导入 / 导出规则

支持规则 JSON 导入导出。

价值：
	•	备份
	•	换设备迁移
	•	分享规则模板（未来）

⸻

9.3 当前页命中原因解释

hover 某个已命中的 tag 或作品时，可以看到：
	•	命中了哪条规则
	•	规则类型是什么
	•	匹配方式是什么

价值：
	•	提升可解释性
	•	降低误伤困惑

⸻

9.4 一键从当前 tag 新建高级规则

当前 MVP 页面内快速添加默认可能是“精确匹配”。
V1.1 可以加一个“高级添加”入口，让用户现场选择：
	•	精确 / 包含 / 通配符
	•	作用范围
	•	规则组

⸻

9.5 临时停用功能

例如：
	•	暂停本页所有规则
	•	暂停某个规则组
	•	暂时显示已折叠作品

价值：
	•	减少用户“被系统接管”的感觉
	•	增强可控性

⸻

Phase 2：规则体系增强版

这一阶段是把产品从“小工具”变成“成熟偏好系统”。

9.6 规则优先级编辑

允许用户定义更细的优先级。

例如：
	•	某条高亮规则优先于某条弱化规则
	•	某条雷点规则只警示，不折叠

⸻

9.7 更丰富的匹配逻辑

例如：
	•	多关键词组合
	•	与 / 或 条件
	•	排除条件
	•	tag 类型限定组合

但这不适合太早做，不然复杂度会暴增。

⸻

9.8 作用范围扩展

规则不仅作用在 tag 本身，还可以作用到：
	•	作品卡片标题
	•	简介 / summary 区域
	•	系列信息（如后续需要）
	•	作者名（谨慎）

⸻

9.9 页面维度规则

例如：
	•	某条规则仅在搜索结果页生效
	•	某条规则仅在某个 fandom 页面生效

价值：
	•	更精细的个性化浏览

⸻

Phase 3：视觉与体验增强版

9.10 多套主题样式

适配不同 AO3 外观风格。

9.11 样式强度配置

例如：
	•	高亮更明显 / 更轻
	•	警示更醒目 / 更克制
	•	噪音弱化程度调节

9.12 更优雅的折叠体验

例如：
	•	批量折叠计数
	•	页面顶部显示“已折叠 X 篇”
	•	一键展开全部

9.13 撤销操作

用户把 tag 加入规则后，可立即撤销。

⸻

Phase 4：高级管理与生态能力

这一阶段不是最优先，但未来可能有价值。

9.14 规则模板

例如：
	•	常见 trope 模板
	•	常见 warning 模板
	•	fandom 社区分享模板

注意：这个阶段才适合讨论分享，不是 MVP。

⸻

9.15 规则统计

例如：
	•	哪条规则命中最多
	•	哪类规则最常用
	•	最近新增规则趋势

价值：
	•	帮助用户理解自己的偏好系统
	•	也方便产品判断真实使用模式

⸻

9.16 跨浏览器迁移

将规则格式抽象后，可逐步支持：
	•	Firefox
	•	Safari（另算）
	•	其他 Chromium 浏览器

⸻

Phase 5：移动端规划预留

这不是现在做，但要提前想清楚未来路线。

9.17 移动端不是直接“兼容”

而是要重做交互层：
	•	hover 改成 tap / long press
	•	菜单改成 action sheet
	•	页面空间更紧凑
	•	折叠提示需要更节制

9.18 当前要预留的只是这些
	•	规则模型不依赖 hover
	•	匹配引擎可复用
	•	样式逻辑可抽离
	•	页面识别层尽量模块化

⸻

10. 推荐版本路线

⸻

V0：验证原型

目标：验证需求和交互

包含：
	•	AO3 模拟页面
	•	hover 按钮
	•	四种操作
	•	假数据反馈

不要求真实插件能力。

⸻

V1：MVP

目标：跑通桌面端 Chrome 闭环

包含：
	•	页面 tag 识别
	•	三种匹配方式
	•	四类规则
	•	页面快速添加
	•	基础规则管理
	•	当前页命中显示
	•	本地存储
	•	折叠 / 警示 / 高亮 / 弱化表现

⸻

V1.1：易用性增强

包含：
	•	导入导出
	•	临时停用
	•	当前页命中原因解释
	•	简单规则分组

⸻

V1.5：规则系统升级

包含：
	•	更细粒度规则管理
	•	页面范围规则
	•	主题适配
	•	统计能力

⸻

V2：跨设备 / 跨端规划

前提是：
	•	桌面端用户留存成立
	•	规则体系稳定
	•	用户确实需要更多平台支持

⸻

11. 最后给一个收敛版结论

如果你现在要真正开做，我建议你把 MVP 只收敛成下面这些：

必须做
	•	桌面 Chrome
	•	支持 AO3 列表页和作品页 tag 识别
	•	支持 relationship / character / freeform
	•	支持精确 / 包含 / 通配符匹配
	•	四类规则：
	•	喜欢：高亮
	•	雷点：警示作品
	•	噪音：弱化 tag
	•	强雷点：折叠作品
	•	tag hover 小按钮
	•	点击菜单快速加规则
	•	插件弹窗查看当前页命中
	•	规则列表管理
	•	本地存储

先别做
	•	移动端
	•	云同步
	•	AI
	•	自动抓取
	•	复杂条件表达式
	•	模板分享系统

最值得优先加的后续功能
	•	导入导出
	•	规则分组
	•	命中原因解释
	•	临时停用
	•	更好的折叠体验
